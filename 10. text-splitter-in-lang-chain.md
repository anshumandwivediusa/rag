# Text Splitter in LangChain

The **Text Splitter** is a utility in LangChain that divides large documents into smaller, manageable chunks. This is essential for RAG (Retrieval-Augmented Generation) pipelines, as embedding models and retrieval systems perform best when processing concise, semantically coherent text segments. By preserving contextual meaning within each chunk, the Text Splitter improves embedding quality, retrieval accuracy, and the relevance of generated responses

## Why Use Text Splitter?
- **Chunking efficiency** → Prevents exceeding token limits in LLMs.  
- **Semantic relevance** → Keeps context meaningful by splitting along logical boundaries.  
- **Improved retrieval** → Smaller chunks increase precision in similarity search.  
- **Flexibility** → Supports multiple splitting strategies (by character, sentence, recursive rules, etc.).

## Types of Text Splitters
| **Splitter Type** | **Description** | **Use Case** |
|-----------------|-----------------|----------------|
| **CharacterTextSplitter** | Splits text by character count. | Simple, fast chunking for uniform text. |
| **RecursiveCharacterTextSplitter** | Splits hierarchically (paragraph → sentence → word). | Preserves semantic boundaries. |
| **TokenTextSplitter** | Splits based on token count (model-specific). | Ensures compatibility with LLM token limits. |
| **MarkdownTextSplitter** | Splits by Markdown structure (headings, lists). | Technical docs, structured notes. |
| **HTMLTextSplitter** | Splits by HTML tags. | Web pages, scraped content. |

## LangChain Text Splitters Explained with Examples

Text splitters are used to break large documents into smaller chunks before creating embeddings and storing them in a vector database. The choice of splitter affects retrieval quality and context preservation.

## 1. CharacterTextSplitter

### How it works

Divides text after a fixed number of characters, regardless of sentence or paragraph boundaries.

### Example

Original Text:

```text
Artificial Intelligence is transforming industries.
Machine Learning is a subset of AI.
Deep Learning is a subset of Machine Learning.
```

Chunk Size = 50 characters

Output:

```text
Chunk 1:
Artificial Intelligence is transforming industries

Chunk 2:
Machine Learning is a subset of AI. Deep Learn

Chunk 3:
ing is a subset of Machine Learning.
```

### Problem

Words and sentences can be cut in the middle.

### When to use

* Quick prototypes
* Uniform datasets
* When semantic preservation is not critical

## 2. RecursiveCharacterTextSplitter

### How it works

Attempts to split at:

```text
Paragraphs (\n\n)
↓
Sentences (\n)
↓
Spaces
↓
Characters
```

Only moves to the next level if the chunk is still too large.

### Example

Input:

```text
Artificial Intelligence is transforming industries.

Machine Learning is a subset of AI.

Deep Learning is a subset of Machine Learning.
```

Chunk Size = 80

Output:

```text
Chunk 1:
Artificial Intelligence is transforming industries.

Chunk 2:
Machine Learning is a subset of AI.

Chunk 3:
Deep Learning is a subset of Machine Learning.
```

### Benefit

Keeps complete ideas together.

### Best Use Case

Most RAG applications use this splitter because it balances:

* Chunk size
* Context preservation
* Retrieval quality

## 3. TokenTextSplitter

### How it works

Splits based on tokens instead of characters.

Tokens are the units LLMs process.

Example:

```text
"ChatGPT is amazing"
```

Approximately:

```text
["Chat", "GPT", " is", " amazing"]
```

≈ 4 tokens

### Example

Suppose:

```text
Max Chunk Size = 100 Tokens
```

Document:

```text
500-token article
```

Output:

```text
Chunk 1 = 100 tokens
Chunk 2 = 100 tokens
Chunk 3 = 100 tokens
Chunk 4 = 100 tokens
Chunk 5 = 100 tokens
```

### Why use it?

LLMs have token limits:

| Model        | Context Window         |
| ------------ | ---------------------- |
| GPT-4o-mini  | Thousands of tokens    |
| GPT-5 family | Larger context windows |

Token splitting ensures chunks fit within model limits.

### Best Use Case

* Production RAG systems
* OpenAI models
* Anthropic models
* Any token-sensitive workflow

## 4. MarkdownTextSplitter

### How it works

Uses Markdown structure to determine chunk boundaries.

Input:

```markdown
# Introduction

LangChain is a framework for LLM applications.

## Text Splitters

Text splitters divide documents into chunks.

## Embeddings

Embeddings convert text into vectors.
```

Output:

```text
Chunk 1:
# Introduction
LangChain is a framework for LLM applications.

Chunk 2:
## Text Splitters
Text splitters divide documents into chunks.

Chunk 3:
## Embeddings
Embeddings convert text into vectors.
```

### Benefit

Preserves document hierarchy.

### Best Use Case

* Documentation
* Knowledge bases
* README files
* Technical guides

---

## 5. HTMLTextSplitter

### How it works

Uses HTML tags as natural boundaries.

Input:

```html
<h1>Introduction</h1>
<p>LangChain is an LLM framework.</p>

<h2>Embeddings</h2>
<p>Embeddings convert text into vectors.</p>
```

Output:

```text
Chunk 1:
Introduction
LangChain is an LLM framework.

Chunk 2:
Embeddings
Embeddings convert text into vectors.
```

### Benefit

Preserves webpage structure.

### Best Use Case

* Website content
* Web scraping pipelines
* CMS exports
* Blogs and articles

## Which Splitter Should You Choose for RAG?

| Scenario                            | Recommended Splitter           |
| ----------------------------------- | ------------------------------ |
| General RAG applications            | RecursiveCharacterTextSplitter |
| PDFs and text files                 | RecursiveCharacterTextSplitter |
| OpenAI/Anthropic production systems | TokenTextSplitter              |
| Technical documentation             | MarkdownTextSplitter           |
| Web scraping projects               | HTMLTextSplitter               |
| Quick proof of concept              | CharacterTextSplitter          |

### Common Production Choice

Most production RAG systems use:

```python
RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200
)
```

because it preserves semantic meaning while generating chunks of manageable size for embeddings and retrieval.

## ⚙️ Key Parameters
- **chunk_size** → Maximum size of each chunk (characters or tokens).  
- **chunk_overlap** → Overlapping text between chunks to preserve context.  
- **separator** → Defines logical split points (e.g., `\n\n`, `.`, `,`).  

## 📚 Best Practices
- Use **RecursiveCharacterTextSplitter** for most cases → balances semantic integrity and token efficiency.  
- Adjust **chunk_size** based on your LLM’s max token limit (e.g., 512–1000 tokens).  
- Apply **chunk_overlap** (10–20%) to avoid losing context between chunks.  
- Choose specialized splitters (Markdown, HTML, etc.) when working with structured documents.

## 🧩 Integration in RAG
The **Text Splitter** is typically the **second step** in a RAG pipeline:  

1. **Document Loader** → Import raw data (PDF, text, web).  
2. **Text Splitter** → Break into chunks.  
3. **Embeddings Model** → Convert chunks into vectors.  
4. **Vector Store** → Store embeddings for similarity search.  
5. **Retriever + LLM** → Answer queries using relevant chunks.
