# Text Splitter in LangChain

## Overview
The **Text Splitter** is a utility in LangChain that breaks large documents into smaller, manageable chunks.  
This is essential for **RAG pipelines**, where embeddings and retrieval models perform best on concise, semantically coherent text segments.

## ✨ Why Use Text Splitter?
- **Chunking efficiency** → Prevents exceeding token limits in LLMs.  
- **Semantic relevance** → Keeps context meaningful by splitting along logical boundaries.  
- **Improved retrieval** → Smaller chunks increase precision in similarity search.  
- **Flexibility** → Supports multiple splitting strategies (by character, sentence, recursive rules, etc.).

---

## 🔧 Types of Text Splitters
| **Splitter Type** | **Description** | **Use Case** |
|-----------------|-----------------|----------------|
| **CharacterTextSplitter** | Splits text by character count. | Simple, fast chunking for uniform text. |
| **RecursiveCharacterTextSplitter** | Splits hierarchically (paragraph → sentence → word). | Preserves semantic boundaries. |
| **TokenTextSplitter** | Splits based on token count (model-specific). | Ensures compatibility with LLM token limits. |
| **MarkdownTextSplitter** | Splits by Markdown structure (headings, lists). | Technical docs, structured notes. |
| **HTMLTextSplitter** | Splits by HTML tags. | Web pages, scraped content. |

---

## ⚙️ Key Parameters
- **chunk_size** → Maximum size of each chunk (characters or tokens).  
- **chunk_overlap** → Overlapping text between chunks to preserve context.  
- **separator** → Defines logical split points (e.g., `\n\n`, `.`, `,`).  

---

## 🚀 Example Usage

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", ".", " "]
)

chunks = text_splitter.split_text(document_text)
```

This produces a list of coherent text chunks ready for embedding and retrieval.

---

## 📚 Best Practices
- Use **RecursiveCharacterTextSplitter** for most cases → balances semantic integrity and token efficiency.  
- Adjust **chunk_size** based on your LLM’s max token limit (e.g., 512–1000 tokens).  
- Apply **chunk_overlap** (10–20%) to avoid losing context between chunks.  
- Choose specialized splitters (Markdown, HTML, etc.) when working with structured documents.  

---

## 🧩 Integration in RAG
The **Text Splitter** is typically the **second step** in a RAG pipeline:  

1. **Document Loader** → Import raw data (PDF, text, web).  
2. **Text Splitter** → Break into chunks.  
3. **Embeddings Model** → Convert chunks into vectors.  
4. **Vector Store** → Store embeddings for similarity search.  
5. **Retriever + LLM** → Answer queries using relevant chunks.
