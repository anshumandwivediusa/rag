# Chunking Strategies 

## Why advanced chunking matters  
  - Because **bi‑encoder embeddings are lossy**, chunking is your first line of defense.  
  - Good chunking → good retrieval → grounded answers.  
  - Bad chunking → hallucinations, missed facts, irrelevant context.

## Bi-encoder vector embeddings are “lossy” by design !

Bi‑encoder embeddings are “lossy” because they compress an entire piece of text—sometimes a paragraph, sometimes a whole document—into a single fixed‑length vector. That compression inevitably throws away detail. Imagine taking a 300‑page book and trying to summarize it into a single sentence: you’ll capture the *general meaning*, but you’ll lose the exact wording, the order of events, the subtle tone shifts, and the small but important facts. Embeddings work the same way. For example, the sentences **“The patient does *not* have diabetes”** and **“The patient has diabetes”** often end up with very similar vectors because the embedding model captures the *topic* (“patient + diabetes”) but loses the critical negation. Or consider a long technical document: if you embed the entire thing into one vector, the model may remember that it’s “about networking protocols,” but it won’t retain the specific detail that **“port 443 must remain open for TLS termination.”** This is why RAG systems sometimes fail to retrieve the right chunk even when the answer is present in the corpus—the embedding simply didn’t preserve the nuance needed to match the query.

This lossy nature also shows up with rare or domain‑specific terms. If your query is **“What is the difference between CYP2D6 and CYP3A4 metabolism?”**, a general‑purpose embedding model might blur both into a generic “pharmacology” vector, making retrieval imprecise. Similarly, embeddings struggle with multi‑step reasoning: if a document says **“A implies B, and B implies C,”** the vector won’t encode that chain of logic. It will just store a fuzzy representation of the topic. Because of this, modern RAG systems increasingly combine embeddings with **hybrid search**, **cross‑encoder reranking**, or **graph‑based retrieval** to recover the detail that the bi‑encoder inevitably loses.

## Chunking Startegies 

Advanced chunking is where weak RAG systems die and strong RAG systems come alive. When you chunk poorly, retrieval becomes noisy, embeddings become meaningless, and the LLM hallucinates because it never sees the right context. When you chunk well, retrieval becomes precise, reranking becomes easier, and the LLM feels “smart” because it finally has the right evidence in front of it.

## 1. **Fixed Chunking**
Splits text into **equal‑sized chunks** (e.g., 200 tokens or 500 characters), regardless of meaning.

**Example:**  
A laptop manual is split every 200 characters:  
- Chunk 1: `"To factory reset the TechCare X15 la..."`  
- Chunk 2: `"ptop, hold the power button until..."`  

**Why it works:**  
It’s **simple and fast**. Useful for prototyping or small datasets. But it may cut sentences in half, reducing semantic clarity.

```
from langchain.text_splitter import CharacterTextSplitter

text = """To factory reset the TechCare X15 laptop, hold the power button until the logo appears.
Then follow the on‑screen instructions to restore factory settings. 
If your battery drains quickly, check power management settings..."""

# Fixed chunking: split every 200 characters
splitter = CharacterTextSplitter(chunk_size=200, chunk_overlap=0)
chunks = splitter.split_text(text)

for i, chunk in enumerate(chunks, 1):
    print(f"Chunk {i}: {chunk}")

```
## 2. **Semantic Chunking**
Instead of splitting text every 500 tokens, you split based on **meaning**.

**Example:**  
A policy document has sections like “Eligibility,” “Exclusions,” “Claims Process.”  
Semantic chunking keeps each section intact instead of slicing mid‑sentence.

**Why it works:**  
Embeddings represent *concepts*, not token boundaries. Keeping coherent meaning improves retrieval dramatically.

```
from langchain.text_splitter import RecursiveCharacterTextSplitter

text = """
Eligibility: Members must be over 18 years old.
Exclusions: Claims related to pre-existing conditions are not covered.
Claims Process: Submit your claim within 30 days of treatment.
"""

# Semantic-aware splitting: respects sentences/paragraphs
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # max size
    chunk_overlap=50,     # overlap for context
    separators=["\n\n", "\n", ".", " "]  # semantic boundaries
)

chunks = splitter.split_text(text)

for i, chunk in enumerate(chunks, 1):
    print(f"Chunk {i}: {chunk}")

```

Output:
  - Chunk 1: "Eligibility: Members must be over 18 years old."

  - Chunk 2: "Exclusions: Claims related to pre-existing conditions are not covered."

  - Chunk 3: "Claims Process: Submit your claim within 30 days of treatment."

## 3. **Recursive Chunking**
Recursive chunking works because you **intentionally create multiple versions of the same document at different levels of granularity**, and each level captures a different kind of meaning. Instead of forcing every query to match a single chunk size (like 500 tokens), you give the retriever a *menu* of options: very small, medium, and large chunks. For example, if you take a 20‑page technical specification and break it into **400 sentence‑level chunks**, **80 paragraph‑level chunks**, and **20 section‑level chunks**, you now have three different “lenses” on the same content. A query like *“What does the API return when authentication fails?”* will likely match a **sentence‑level chunk**, because the answer is usually a single line like *“The API returns HTTP 401 for invalid credentials.”* But a query like *“Explain the full authentication flow”* needs a **section‑level chunk**, because the answer spans diagrams, steps, and multiple paragraphs. And a query like *“What are the design principles behind this system?”* might match a **paragraph‑level chunk**, because the answer is conceptual but not long.

This multi‑granularity approach solves a core problem of bi‑encoder embeddings: they are lossy and cannot represent all details in a single vector. By giving the retriever multiple chunk sizes, you reduce the chance that the “right” information is trapped inside a chunk that is too big (and therefore too fuzzy) or too small (and therefore missing context). In practice, recursive chunking dramatically improves retrieval accuracy because the system can choose the chunk size that best matches the **semantic scope** of the query. This is why modern RAG systems treat recursive chunking as a foundational technique rather than an optional optimization.

You create **multiple granularities** of chunks:  
- Small chunks (Sentences)  
- Medium chunks (Paragraphs)  
- Large chunks (Sections)

**Example:**  
A 20‑page technical spec becomes:  
- 400 sentence‑level chunks  
- 80 paragraph‑level chunks  
- 20 section‑level chunks  

Retrieval can then pick the right level of detail.

**Why it works:**  
Some queries need a sentence.  
Some need a whole section.  
Recursive chunking gives the retriever options.


```sql
CREATE TABLE techcare_chunks (
    id SERIAL PRIMARY KEY,
    document_id INTEGER,          -- original document reference
    chunk_type TEXT,              -- 'sentence', 'paragraph', 'section'
    chunk_index INTEGER,          -- order within the document
    chunk_text TEXT,              -- the actual chunk
    metadata JSONB,               -- optional
    embedding_vector VECTOR(768)  -- embedding for this chunk
);
```

```sql
WITH ranked AS (
    SELECT
        id,
        document_id,
        chunk_type,
        chunk_index,
        chunk_text,
        metadata,
        embedding_vector <-> '[0.12, -0.44, 0.91, ...]'::vector AS distance
    FROM techcare_chunks
    ORDER BY distance
    LIMIT 20
)
SELECT *
FROM ranked
ORDER BY
    CASE chunk_type
        WHEN 'sentence' THEN 1
        WHEN 'paragraph' THEN 2
        WHEN 'section' THEN 3
        ELSE 4
    END,
    distance;
```

## 3. **Hierarchical Chunking**
Hierarchical chunking builds a tree of chunks at multiple levels of abstraction, so the retriever can match the scope of the user’s question. At the top of the tree, you store a document summary that captures the big picture. Below that, you store section summaries, then full paragraphs, and finally individual sentences. For example, if a user asks “What is the overall purpose of this system”, the retriever should surface the top‑level summary, because the question is broad. But if the user asks “How does the authentication handshake work”, the retriever should dive deeper into the tree and return a sentence‑ or paragraph‑level chunk from the authentication section. This mirrors how humans skim documents: we start with the overview, then zoom into the relevant part. Hierarchical chunking gives the retriever the same ability to “zoom,” which dramatically improves accuracy for both high‑level conceptual questions and low‑level technical ones.

You build a **tree** of chunks:  
- Level 1: Document summary  
- Level 2: Section summaries  
- Level 3: Paragraphs  
- Level 4: Sentences  

**Example:**  
A query like “What is the overall purpose of this system?” retrieves the top-level summary.  
A query like “How does the authentication handshake work?” retrieves a deep node.

**Why it works:**  
It mirrors how humans skim documents.

```sql
CREATE TABLE rag_hier_chunks (

    parent_id INTEGER,                     -- points to the parent chunk
```

## 4. **Entity‑Aware Chunking**
Chunks are created around **entities** (people, chemicals, APIs, legal terms).

**Example:**  
In a medical corpus, you chunk around:  
- “CYP2D6 metabolism”  
- “CYP3A4 inhibitors”  
- “Warfarin dosing guidelines”

This ensures that rare, domain‑specific terms stay intact.

**Why it works:**  
Embeddings struggle with rare terms; entity‑aware chunking preserves them.

## 5. **Graph‑Based Chunking**
You build a **knowledge graph** first, then chunk based on relationships.

**Example:**  
In a software architecture doc:  
- “Service A → Service B → Database C”  
becomes a connected chunk group.

**Why it works:**  
Graph structure preserves relationships that embeddings lose.


## 6. **Query‑Aware Chunking**
Chunks are generated **dynamically** based on the user’s query.

**Example:**  
If the query is:  
“Explain how OAuth refresh tokens are rotated,”  
the system extracts only the relevant spans from multiple documents and merges them into a custom chunk.

**Why it works:**  
You don’t rely on static chunks at all — you build the chunk *on demand*.



## 7. **Sliding Window Chunking**
Chunks overlap to preserve context.

**Example:**  
Chunk 1: tokens 0–500  
Chunk 2: tokens 400–900  
Chunk 3: tokens 800–1300  

**Why it works:**  
Prevents losing meaning at boundaries (e.g., definitions split across chunks).



## 8. **Structure‑Preserving Chunking**
You chunk based on **document structure**:  
- headings  
- bullet lists  
- tables  
- code blocks  

**Example:**  
A code file is chunked by function, not by token count.  
A legal contract is chunked by clause.

**Why it works:**  
Structure carries meaning that embeddings alone cannot capture.



## 9. **Hybrid Chunking**
Combine multiple strategies:  
- semantic + sliding window  
- entity‑aware + hierarchical  
- recursive + structure‑preserving  

**Example:**  
A pharmaceutical RAG system might use:  
- entity‑aware chunking for drug names  
- semantic chunking for clinical guidelines  
- sliding windows for research papers  

**Why it works:**  
Different content types need different chunking logic.

- Chunking for long‑context models  
- Chunking for code vs legal vs medical documents  
- How chunking interacts with reranking
