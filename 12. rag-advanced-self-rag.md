# Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection

*Asai et al., University of Washington & IBM Research AI — ICLR 2024 (Oral, Top 1%)*

---

## The Core Problem

Standard LLMs are prisoners of their parametric knowledge — they can only answer from what was baked into their weights during training. Traditional RAG attempts to escape this by prepending retrieved passages to every prompt, but this introduces a subtler pathology: **indiscriminate retrieval**. Fetching passages unconditionally, regardless of whether a query actually needs external grounding, can inject irrelevant or off-topic context and actively degrade generation quality. Moreover, even when relevant passages are retrieved, the output is not guaranteed to be consistent with retrieved relevant passages, since the models are not explicitly trained to leverage and follow facts from provided passages.

Self-RAG addresses both failure modes simultaneously.

---

## The Central Idea: Reflection Tokens

Self-RAG trains a single arbitrary LM in an end-to-end manner to reflect on its own generation process by generating both task output and intermittent special tokens — called **reflection tokens**. Reflection tokens are categorized into **retrieval** and **critique** tokens to indicate the need for retrieval and the quality of generation respectively.

This is the conceptual leap: rather than retrieval being an external pipeline step, it becomes **a learned behavior emergent from the model itself**. The model decides when to retrieve, evaluates what it retrieves, generates a response, and then critiques its own output — all within a single unified generation pass.

There are four types of reflection tokens, each answering a distinct question:

| Token | Question Asked |
|---|---|
| `Retrieve` | Should I fetch external passages at all? |
| `ISREL` | Is the retrieved passage relevant to the query? |
| `ISSUP` | Is my generated output actually *supported* by the passage? |
| `ISUSE` | How useful is this response overall (1–5)? |

Generating reflection tokens makes the LM controllable during the inference phase, enabling it to tailor its behavior to diverse task requirements.

---

## The Inference Algorithm

At each generation step, the model follows a principled decision loop:

1. **Retrieval Decision** — The model predicts a `Retrieve` token (`yes / no / continue`). If no retrieval is needed (e.g., a creative writing prompt), it simply generates the next segment.
2. **Parallel Evidence Evaluation** — If retrieval is triggered, multiple passages are retrieved and processed *in parallel*. For each passage, the model predicts `ISREL` — filtering out irrelevant ones before they can corrupt the generation.
3. **Grounded Generation** — The model generates candidate response segments, one per passage.
4. **Self-Critique and Ranking** — `ISSUP` checks whether the segment is fully, partially, or not supported by its passage. `ISUSE` scores overall response quality. Candidates are ranked by a weighted combination of these scores, and the best segment is selected.

This process differs from conventional RAG, which consistently retrieves a fixed number of documents regardless of retrieval necessity and never revisits generation quality. Moreover, Self-RAG provides citations for each segment with its self-assessment of whether the output is supported by the passage, leading to easier fact verification.

---

## How It's Trained: A Two-Model System

Self-RAG requires training two models: a **Critic** and a **Generator**.

**The Critic Model (C):** Manual annotation of reflection tokens for each segment is expensive. A state-of-the-art LLM like GPT-4 can be effectively used to generate such feedback. The authors collect 4k–20k supervised training data for each reflection token type by prompting GPT-4, then distill this knowledge into an in-house critic model. The critic achieves >90% agreement with GPT-4 predictions on most token categories — meaning the distilled model captures GPT-4's judgment without requiring it at inference time.

**The Generator Model (M):** The generator is trained on a curated corpus with interleaving passages retrieved by a retriever R and reflection tokens predicted by the critic model C. Reflection tokens are inserted into the original corpus *offline* by the trained critic. This eliminates the need to host a critic model during training, reducing overhead. The generator then learns to emit these tokens naturally as part of next-token prediction — the reflection tokens simply become part of the model's expanded vocabulary.

---

## Why This Matters Conceptually

The key insight Self-RAG encodes is that **retrieval is not always helpful**, and **generation quality is not self-evident**. Most RAG pipelines treat these as assumptions; Self-RAG treats them as learned variables. This has several architectural implications:

- **Adaptive retrieval** means the model can handle queries ranging from those requiring zero external knowledge to those needing multiple retrieval steps within a single response.
- **Self-critique** decouples generation from evaluation — the model develops an internal auditor that runs alongside its generative process.
- **Controllable inference** — because behavior is governed by reflection token probabilities, users can tune retrieval frequency, trade off citation precision vs. completeness, or enforce hard constraints at test time without retraining.

The inference-time algorithm enables flexibly adjusting retrieval frequency for different downstream applications and customizing model behaviors to user preferences by leveraging reflection tokens through segment-level beam search using a weighted linear sum of the reflection token probabilities as the segment score.

---

Self-RAG is less a retrieval system and more a framework for making a model **epistemically self-aware** — teaching it to know what it knows, know when it doesn't, and verify whether what it says is actually grounded. That's the conceptual core.
