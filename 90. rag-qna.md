# Questions and Answers

### What is relational DB?
**Relational Database (RDB):** A database that stores data in **tables (rows and columns)** and uses **relationships between tables** to organize, retrieve, and maintain data efficiently.
Codd's 13 Rules (Rule 0–12)

| **Rule No.** | **Rule Name**                              | **Description**                                                                                           | **Example**                                                                                     |
| ------------ | ------------------------------------------ | --------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| **Rule 0**   | Foundation Rule                            | The DBMS must manage all data using only relational capabilities.                                         | All data is stored and accessed through relational tables.                                      |
| **Rule 1**   | Information Rule                           | All information must be represented as values in rows and columns of tables.                              | Employee data is stored in an `Employee` table with columns like `EmpID`, `Name`, and `Salary`. |
| **Rule 2**   | Guaranteed Access Rule                     | Every data value must be accessible using the table name, primary key, and column name.                   | Retrieve salary using `EmpID` and the `Salary` column.                                          |
| **Rule 3**   | Systematic Treatment of NULL Values        | The DBMS must support NULL values to represent missing, unknown, or inapplicable information.             | An employee without a manager has `ManagerID = NULL`.                                           |
| **Rule 4**   | Dynamic Online Catalog                     | Database metadata (schema) must be stored as relational tables and be queryable.                          | Query `INFORMATION_SCHEMA.TABLES` to view table definitions.                                    |
| **Rule 5**   | Comprehensive Data Sublanguage Rule        | The DBMS must provide a single language for defining, querying, updating, and controlling data.           | SQL supports `CREATE`, `SELECT`, `INSERT`, `UPDATE`, and `DELETE`.                              |
| **Rule 6**   | View Updating Rule                         | All theoretically updatable views should be updatable by the DBMS.                                        | Updating an employee's salary through a simple view updates the base table.                     |
| **Rule 7**   | High-Level Insert, Update, and Delete Rule | The DBMS must support set-based operations on multiple rows.                                              | `UPDATE Employee SET Salary = Salary * 1.10;` updates all employees at once.                    |
| **Rule 8**   | Physical Data Independence                 | Changes to physical storage should not affect applications.                                               | Creating an index or moving data to another disk does not require application changes.          |
| **Rule 9**   | Logical Data Independence                  | Changes to the logical schema should minimally impact applications.                                       | Adding an `Email` column does not break existing queries.                                       |
| **Rule 10**  | Integrity Independence                     | Integrity constraints must be stored and enforced by the DBMS, not by application code.                   | Primary keys, foreign keys, and `CHECK` constraints enforce data validity.                      |
| **Rule 11**  | Distribution Independence                  | Applications should work the same regardless of whether data is stored on one server or multiple servers. | A distributed database appears as a single database to users.                                   |
| **Rule 12**  | Non-Subversion Rule                        | No low-level interface should bypass integrity constraints or security rules.                             | Direct file access cannot violate primary key or foreign key constraints.                       |

## RDBMS vs Graph vs Vector DB?

| **Feature**                | **RDBMS**                           | **Graph Database**                                       | **Vector Database**                                         |
| -------------------------- | ----------------------------------- | -------------------------------------------------------- | ----------------------------------------------------------- |
| **Conceptual Meaning** | **"What data do I have?"** – Organizes structured information into tables with rows and columns. Normal Relationship | **"How is the data connected?"** – Models entities and the relationships between them as a graph. Deep nested meaning | **"What data is semantically similar?"** – Represents data as embeddings and finds information based on meaning rather than exact values. Similarity|
| **Data Model**             | Tables (rows & columns)             | Nodes and Relationships (Edges)                          | High-dimensional vectors (embeddings)                       |
| **Primary Purpose**        | Store structured data               | Store and query connected data                           | Store and retrieve embeddings for semantic search           |
| **Relationship Handling**  | Foreign Keys and Joins              | Native graph relationships                               | Based on vector similarity, not explicit relationships      |
| **Query Language**         | SQL                                 | Cypher, Gremlin, SPARQL                                  | Vector similarity APIs, SQL extensions (varies by database) |
| **Search Type**            | Exact match, filtering, joins       | Relationship traversal                                   | Semantic similarity search                                  |
| **Best For**               | Business applications, transactions | Social networks, fraud detection, recommendation engines | RAG, semantic search, AI applications                       |
| **Performance**            | Excellent for structured queries    | Excellent for traversing complex relationships           | Excellent for nearest-neighbor similarity search            |
| **Scalability**            | High                                | High                                                     | High (optimized for vector indexing)                        |
| **Data Structure Example** | Customer, Orders, Products          | People connected by friendships                          | Text embeddings, image embeddings                           |
